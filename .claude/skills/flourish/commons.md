# Skill: commons

## Purpose
The instinct toward shared ownership is often right but can also be forced. This skill helps a community think through a practical decision concretely — based on how often people actually need something, who can maintain it, and what model genuinely fits this group. An individual purchase is sometimes the right answer; the skill honors that.

## Trigger
- Explicit: `/commons power tools`
- `/commons car`
- `/commons [any item or category the group is deciding about]`

---

## Ownership Models Considered

```
MODEL A — Shared ownership
  The group owns one item together.
  Requires: storage steward, access protocol, maintenance agreement

MODEL B — Lending library
  Members share what they individually own.
  Catalog of who has what. Borrowing is informal but tracked.

MODEL C — External borrowing
  Library tool loan programs, rental, neighbor borrowing
  No ownership cost. Access is occasional and accepted as such.

MODEL D — Individual purchase
  Each person/household owns their own.
  Right when: high frequency use, personal fit matters, usage overlaps significantly
```

---

## Steps

```
1. DEFINE WHAT'S ACTUALLY BEING CONSIDERED
   "What does everyone actually need from [item]?
    How often do you each use something like this?
    In what situations?"

2. MAP USAGE (collect from each member):
   Build a simple table:
   
   | Member | Frequency needed | Duration per use | Storage capacity | Notes |
   
   → This often reveals that the need is much smaller than imagined
   → Or that usage is genuinely overlapping and sharing would be hard

3. PRESENT THE 4 MODELS briefly
   "There are a few ways communities handle this.
    I'll describe them quickly — then let's see what fits."
   [Brief descriptions of A/B/C/D]

4. EXPLORE FIT — one model at a time, not a vote
   "Let's look at [most likely model] first.
    What would actually be required to make that work?
    Who would steward it? Where would it live? What happens when it's broken?"
   → Practical questions surface real blockers. Let them emerge.

5. SURFACE THE TRADEOFFS HONESTLY
   "Shared ownership means: [concrete pros]. It also means: [concrete cons].
    Is this group set up for that?"
   → Don't sell any model. Just illuminate it.

6. IF GROUP REACHES A DECISION:
   → If commons model: draft a simple agreement
     "Steward: [person]. Stored at: [location].
      To borrow: [method]. Maintenance responsibility: [shared / steward].
      Review this in: [timeframe]."
   → If individual purchase: "That makes sense given [reason]. No further action needed."

7. IF GROUP CAN'T DECIDE:
   "What information would help? Or does this need more time?"
   → Don't force resolution. A delayed decision is often the right decision.

OUTPUTS:
- Commons agreement saved to memory/community_commons.md (if decided)
- Usage mapping table saved regardless of outcome

GUARDS:
- Individual purchase is NEVER a failure — don't treat it as one
- Don't push toward sharing if the usage data doesn't support it
- If the item in question is personal in nature (clothing, medical, hygiene):
  gently note: "Some things don't lend well — this might be one."
- If one member would carry disproportionate burden in any model:
  name it directly: "It looks like [aspect] would fall mostly on [person].
  Is that something they're genuinely okay with?"
```
